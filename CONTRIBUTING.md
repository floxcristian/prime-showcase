# Contribuir a `prime-showcase`

Esta guía describe cómo configurar el entorno, cómo proponer cambios, qué
disciplina seguimos en commits y PRs, y cuándo un cambio escala a RFC o
requiere aprobación del design council. No es decorativa: el repo se
evalúa contra estos criterios en cada revisión.

Documentos hermanos:

- [`README.md`](./README.md) — overview del stack.
- [`AGENTS.md`](./AGENTS.md) — guía operativa para agentes y devs.
- [`DESIGN.md`](./DESIGN.md) — identidad visual, tokens, recetas.
- [`docs/GOVERNANCE.md`](./docs/GOVERNANCE.md) — quién decide qué.
- [`docs/RFC_TEMPLATE.md`](./docs/RFC_TEMPLATE.md) — plantilla de RFC.
- [`docs/DEPRECATION.md`](./docs/DEPRECATION.md) — política de deprecación.
- [`CHANGELOG.md`](./CHANGELOG.md) — historial de cambios y bumps de tokens.

## Setup local

Requisitos: **Node 22** (matchea CI, ver `.github/workflows/ci.yml`) y
npm 10+. Recomendamos usar `nvm` o `volta` para fijar la versión.

```bash
git clone <repo-url> prime-showcase
cd prime-showcase
npm install                 # corre `postinstall` → patch-package aplica patches/
npm start                   # dev server en :4200 (Angular dev server + SSR)
npm run storybook           # catálogo de componentes en :6006
```

`npm install` ejecuta `patch-package` automáticamente. Si ves errores al
arrancar relativos a PrimeNG, verificá que `patches/primeng+21.1.5.patch`
se aplicó (ver `docs/adr/001-ssr-hydration-and-primeng-theming.md` §6).

## Workflow de branches

Las ramas siguen un prefijo que indica intención. CI no falla si no lo
respetás, pero los reviewers lo piden:

| Prefijo | Para qué | Ejemplo |
|---|---|---|
| `feature/` | Funcionalidad nueva visible al usuario | `feature/inbox-bulk-actions` |
| `fix/` | Bugfix sin cambio de API ni de diseño | `fix/chart-tooltip-clipping` |
| `chore/` | Mantenimiento, dependencias, refactor sin cambio funcional | `chore/bump-angular-21.3` |
| `docs/` | Cambios solo en `*.md`, `docs/`, `AGENTS.md`, `DESIGN.md` | `docs/contributing-rfc-flow` |
| `design-tokens/` | Cualquier toque a `src/app/app.preset.ts` o `design-tokens/tokens.json` | `design-tokens/primary-hue-shift` |
| `rfc/` | RFC en `docs/rfc/NNNN-titulo.md` (estado draft o accepted) | `rfc/0002-density-toggle` |
| `eslint/` | Nuevas reglas o cambios al plugin local | `eslint/no-decorative-pi-icon` |

Una sola unidad de cambio por branch. Si arrancás un `feature/` y
detectás un bug ortogonal, abrí un `fix/` aparte y mergéalo primero.

## Convenciones de commit

Usamos commits estilo Conventional Commits, **sin tipo obligatorio de
scope pero altamente recomendado**. El scope va entre paréntesis y
referencia la feature, primitivo, regla ESLint o capa afectada.

```text
<type>(<scope>): <imperativo en minúscula, < 72 chars>

<cuerpo opcional, wrap a 72 chars, explicar QUE y POR QUE>

<footer opcional: Refs #123, Closes #456, BREAKING CHANGE: ...>
```

Tipos aceptados:

- `feat` — funcionalidad nueva (entra en CHANGELOG `### Added`).
- `fix` — bugfix (entra en CHANGELOG `### Fixed`).
- `chore` — mantenimiento sin impacto en producto.
- `docs` — solo documentación.
- `refactor` — refactor sin cambio de comportamiento.
- `perf` — mejora de performance medible (citar bundle:check si aplica).
- `test` — agregar / corregir tests.
- `tokens` — cambios en `app.preset.ts` o `design-tokens/tokens.json`
  (entran en CHANGELOG `### Tokens`).
- `style` — cambios visuales en templates sin tocar tokens.
- `build` — Angular config, webpack, postcss, etc.
- `ci` — `.github/workflows/*`.

Ejemplos válidos del repo:

```text
feat(toast): unify position to bottom-right across modules
fix(motion): drop transition-all from carousel arrow hover
tokens(palette): retune primary.500 to #0074C2 to match logo
docs(design): split DESIGN.md into autogenerated + manual sections
refactor(eslint): extract createClassAttrVisitor to utils
chore(deps): bump primeng to 21.1.5
```

`BREAKING CHANGE:` en el footer es obligatorio cuando renombrás un token
público, removés una clase exportada o cambiás el shape de un primitivo
de `src/app/shared/components/`. Ver
[`docs/DEPRECATION.md`](./docs/DEPRECATION.md) para el flujo completo.

## Antes de cada commit

Mínimo:

```bash
npm run lint                 # ng lint + RuleTester + design-tokens:check
npm run design-tokens:check  # ya es parte de `lint`, listado por claridad
```

Para cambios que toquen runtime, build o SSR:

```bash
npm run verify               # lint + build + bundle:check + smoke (paridad con CI)
```

Si modificaste `src/app/app.preset.ts`:

```bash
npm run design-tokens:sync   # regenera DESIGN.md + design-tokens/tokens.json
git add DESIGN.md design-tokens/tokens.json src/app/app.preset.ts
```

Commiteá los tres juntos. CI corre `design-tokens:check` (sin `--update`)
y falla si detecta drift. Ver `tools/design-tokens/sync.mjs` para el
mecanismo de patch quirúrgico sobre DESIGN.md.

## Tests

- **Unit tests** (`*.spec.ts`) — vía `npm test` (vitest). Mínimo: cubrir
  lógica no trivial de servicios, computed signals, pipes y directivas.
  Componentes visuales pueden omitirse si están cubiertos por Storybook +
  Playwright visual.
- **ESLint plugin tests** (`tools/eslint/rules/__tests__/*.test.js`) —
  obligatorio para cada regla nueva. Ver `.claude/rules/eslint-plugin.md`
  para el patrón con `RuleTester`.
- **Design tokens resolver tests**
  (`tools/design-tokens/__tests__/*.test.mjs`) — obligatorio si tocás
  `tools/design-tokens/resolver.mjs`.
- **SSR smoke** (`tools/smoke/ssr-theme.smoke.mjs`) — cubre 4 casos de
  cookie del dark-mode SSR. Corre vía `npm run test:ssr:smoke` con el
  server local en :4000, o vía `npm run verify` con auto-server.
- **Visual regression** (`tests/visual/*.spec.ts`) — Playwright
  `toHaveScreenshot()` a nivel ruta y a nivel componente Storybook.
  Cambios visuales intencionales requieren refrescar baselines vía el
  workflow `Visual baselines (manual)` (preferido sobre `npm run
  visual:update` local, que puede capturar diferencias de fuente del
  host).
- **A11y** (`npm run a11y`) — axe-core scan vía Playwright. Falla en
  cualquier violación `serious` o `critical`. Corre en `.github/workflows/a11y.yml`.

## Pull Requests

### Tamaño y alcance

Apuntá a PRs **chicas y atómicas**. Reviewers humanos pierden contexto a
partir de ~400 líneas modificadas. Si un cambio es grande, partilo en una
serie de PRs encadenados con commits independientes que cada uno deje el
build verde.

### Descripción

Toda PR debe responder tres preguntas en su descripción:

1. **¿Qué cambia?** — una línea por unidad de cambio.
2. **¿Por qué?** — link a issue, RFC, o explicación inline.
3. **¿Cómo se verifica?** — checklist de smoke manual + screenshots si
   hay cambio visual.

Para PRs visuales: pegar **antes/después** lado a lado (capturas claras y
oscuras). Para PRs de tokens: pegar el diff de `design-tokens/tokens.json`
en el cuerpo.

### Checklist de code review

El reviewer evalúa contra estos criterios (en orden de prioridad). Si una
PR falla en cualquiera, va a changes-requested:

**Arquitectura — bloqueantes**

- [ ] PrimeNG primero: no se creó un wrapper custom si PrimeNG tiene el
      componente. Si se creó: justificación en la descripción.
- [ ] Componentes con `ChangeDetectionStrategy.OnPush` y `standalone:
      true`. Sin excepciones.
- [ ] Estado con `signal()` + `computed()` + `effect()`. Nada de
      `BehaviorSubject` para estado de UI local.
- [ ] DI con `inject()`, no via constructor.
- [ ] Pipes Angular (`| date`, `| number`) ausentes en templates —
      formateo en `.ts` con `Intl.*` y `computed()`.

**Diseño — bloqueantes**

- [ ] Solo design tokens (`text-color`, `bg-surface-*`, `bg-primary`). Sin
      `text-gray-*`, `bg-blue-*`, hex hardcodeado.
- [ ] Tailwind en template; `.scss` del componente queda vacío.
- [ ] Cada `cursor-pointer` tiene su `hover:*` correspondiente y viceversa
      (`showcase/hover-requires-cursor-pointer` lo enforce, pero verificá
      visualmente).
- [ ] Cada `bg-surface-*` tiene su `dark:bg-surface-*`
      (`showcase/no-missing-dark-pair`).
- [ ] Tipografía dentro de las recetas de [`DESIGN.md`](./DESIGN.md).
      Combinaciones nuevas requieren actualizar DESIGN.md primero.
- [ ] Sin `transition-all`. Solo `transition-colors`, `transition-opacity`,
      `transition-transform` o `transition-[transform]`.
- [ ] Sin `shadow-*` (excepto los dos casos documentados en `DESIGN.md`
      sección "Elevation & Depth").
- [ ] Spacing y `rounded-*` dentro de la escala permitida.

**Accesibilidad — bloqueantes**

- [ ] Icon-only `<p-button>` lleva `aria-label` + `pTooltip`
      (`showcase/no-icon-button-without-tooltip`).
- [ ] Iconos decorativos llevan `aria-hidden="true"`
      (`showcase/no-decorative-icon-without-aria-hidden`).
- [ ] Touch targets de pointers gruesos llegan a 44x44px (WCAG 2.5.5).
- [ ] Contenido que cambia en runtime tiene `aria-live` apropiado
      (`polite` por default, `assertive` solo para errores críticos).
- [ ] Dialogs y menús: focus trap, navegación con flechas, cierre con
      Esc. Verificar manualmente además del axe-core scan.

**Performance — recomendados**

- [ ] Bloques pesados above-the-fold no se mueven (CLS = 0). Cuando hay
      carga async, usar `<p-skeleton>` con dimensiones reales.
- [ ] Bloques pesados below-the-fold considerar `@defer (hydrate on
      viewport)` (ver `.claude/rules/ssr-and-runtime.md`).
- [ ] `bundle:check` no detecta crecimiento > 3% del bundle inicial.

**Docs y tokens — recomendados**

- [ ] Si modificaste `app.preset.ts`: `npm run design-tokens:sync` corrido
      y los tres archivos commiteados juntos.
- [ ] Si agregaste un primitivo a `src/app/shared/components/`: hay
      story correspondiente en `src/stories/primitives/`.
- [ ] Si agregaste una receta compuesta: story en `src/stories/recipes/`.
- [ ] Cambios visibles al usuario reflejados en `CHANGELOG.md`
      `## [Unreleased]` con la subsección correcta.

### Approvals requeridas

| Tipo de cambio | Reviewers | Quién aprueba final |
|---|---|---|
| Bugfix simple, < 50 LOC, sin cambio de API | 1 contributor | Cualquier maintainer |
| Feature pequeña, sin RFC | 1 contributor + 1 maintainer | Maintainer |
| Cambio en `src/app/app.preset.ts` (tokens) | 1 maintainer + 1 design council | RFC accepted + design council |
| Nuevo primitivo en `src/app/shared/components/` | 1 maintainer + 1 design council | RFC accepted |
| Nueva regla ESLint en `tools/eslint/rules/` | 1 maintainer + 1 design council | RFC accepted |
| Cambio en governance docs (`GOVERNANCE.md`, `CONTRIBUTING.md`, `DEPRECATION.md`) | 2 design council | Quórum design council |
| Cambio en `app.config.ts`, `app.routes.ts`, SSR setup | 2 maintainers | Maintainer + nota en ADR |

Detalle en [`docs/GOVERNANCE.md`](./docs/GOVERNANCE.md).

## Cuándo se requiere un RFC

Un RFC (Request For Comments) es un documento markdown en
`docs/rfc/NNNN-titulo.md` siguiendo la plantilla en
[`docs/RFC_TEMPLATE.md`](./docs/RFC_TEMPLATE.md). Se requiere RFC para:

1. **Cambios en `src/app/app.preset.ts`** — palette, motion tokens,
   z-index, density, focus ring, component overrides. El preset es API
   pública para Storybook, app y consumers externos (tokens.json).
2. **Nuevos primitivos en `src/app/shared/components/`** — agregar un
   nuevo componente compartido entre features. Excluye refactors de
   primitivos existentes (eso va por PR + 1 maintainer).
3. **Nuevas reglas ESLint en `tools/eslint/rules/`** — el plugin local es
   parte del contrato del design system. Cada regla nueva amplía o
   restringe lo que el proyecto considera "código válido".
4. **Cambios en documentos de governance** —
   [`CONTRIBUTING.md`](./CONTRIBUTING.md),
   [`GOVERNANCE.md`](./docs/GOVERNANCE.md),
   [`DEPRECATION.md`](./docs/DEPRECATION.md),
   [`AGENTS.md`](./AGENTS.md).
5. **Cambios arquitecturales** — nueva dependencia mayor, cambio de SSR
   strategy, migration de Angular major, cambio en el tree de routing.
   Estos además requieren ADR en `docs/adr/`.
6. **Deprecación de cualquier API pública** — clase, primitivo, token,
   regla ESLint. Ver [`docs/DEPRECATION.md`](./docs/DEPRECATION.md).

NO requieren RFC:

- Bugfixes en componentes existentes.
- Refactors internos sin cambio de API.
- Nuevas features dentro de un `src/app/modules/` que solo usan APIs
  públicas ya existentes.
- Documentación, tipos, tests.

### Flujo de RFC

1. Crear `docs/rfc/NNNN-titulo.md` desde la plantilla, con `status:
   draft` y un placeholder en `rfc:` que será asignado al merge.
2. Abrir PR con prefijo `rfc/` y label `rfc:draft`.
3. Discusión en la PR mínimo 5 días hábiles. El autor responde a comments
   editando el RFC, no debatiendo en threads.
4. Para aceptar: cambiar `status: accepted`, asignar número definitivo
   (siguiente disponible), 2 approvals de design council, merge.
5. Implementación va en PRs separadas, cada una referenciando el RFC en
   el footer del commit (`Refs RFC-NNNN`).

Un RFC se puede `rejected` (con explicación en el doc) o `superseded` por
otro RFC más adelante. Los RFCs viejos se quedan en el repo como
historia.

## `/ultrareview`

Para code review profundo multi-agente del branch actual o un PR
existente, el usuario puede invocar `/ultrareview` o `/ultrareview <PR#>`
en Claude Code. Es user-triggered y billed — los agentes no pueden
lanzarlo por su cuenta. Cuando un reviewer pide "revisar el PR con más
profundidad", recordá que `/ultrareview` existe y es la opción correcta
sobre `/review` (que es lightweight).

`/ultrareview` cubre lo que un `/review` lightweight no alcanza:
arquitectura cruzada entre módulos, validación contra DESIGN.md y AGENTS.md,
audit de patrones contra la base existente. No reemplaza review humano —
lo amplifica.

## Reportar bugs

Issues en el repo. No hay templates formales en
`.github/ISSUE_TEMPLATE/` todavía (issue abierto: agregar templates es
trabajo de un PR futuro etiquetado `chore/issue-templates`). Mientras
tanto, todo issue debe incluir:

```text
**Descripción**
Una frase: qué pasa y qué esperabas.

**Pasos para reproducir**
1. Navegar a /...
2. Click en ...
3. Observar ...

**Comportamiento esperado**
...

**Comportamiento actual**
...

**Capturas**
(arrastrar imagen o pegar URL)

**Contexto**
- Navegador y versión:
- Tema (claro / oscuro):
- Resolución:
- Rama y commit SHA:
```

Para bugs de accesibilidad incluir además: screen reader usado, output
del axe-core scan (`npm run a11y` local), si afecta navegación por
teclado.

Para regresiones visuales en CI: link al run de `Visual regression`
fallido + adjuntar el diff PNG que el workflow expone como artifact.

## Reportar vulnerabilidades de seguridad

Vulnerabilidades NO van en issues públicas. Mandar email al equipo de
seguridad (ver `SECURITY.md` cuando exista — issue abierto:
`docs/security-policy`). Mientras tanto, contactar a un maintainer por
canal privado del equipo.

## Estilo de código

- **Idioma:** identifiers en inglés. Comments, commits, docs y PR
  descriptions en español (matchea el resto del repo). Excepciones:
  términos técnicos universales (RFC, ADR, semver, SSR, CLS) quedan en
  inglés sin traducir.
- **TypeScript:** strict mode. `strictTemplates`, `noImplicitOverride`,
  `noPropertyAccessFromIndexSignature`. Ya enforced en `tsconfig.json`.
- **Imports:** path-relative para imports intra-feature, alias absolutos
  cuando atraviesan features. Sin imports circulares.
- **Naming:** PascalCase para classes/interfaces/types/enums, camelCase
  para variables/funciones/props, UPPER_SNAKE_CASE para constantes
  exportadas, kebab-case para archivos y selectores.
- **Un export por archivo** para constantes y mocks. Componentes y
  servicios pueden coexportar tipos relacionados.

## Acuerdos de versionado

El `package.json` se mantiene en `version: 0.0.0` porque el proyecto no
se publica a npm. Sin embargo, los **tokens del design system tienen
versionado semántico** trackeado en [`CHANGELOG.md`](./CHANGELOG.md):

- **Major** — cambio breaking: rename de un token, cambio semántico de un
  primary.500 sin alias de compat, remoción de una clase exportada por
  un primitivo. Requiere ADR + migration path.
- **Minor** — feature aditiva sin breaking: nuevo token, nuevo primitivo,
  nueva regla ESLint, deprecación de algo (que sigue funcionando).
- **Patch** — bugfix, refinamiento sin cambio visual significativo,
  ajustes de docs.

Los `[Unreleased]` se cortan a release nombrado cuando un maintainer
ejecuta el workflow de release (aún no automatizado — issue abierto:
`chore/release-automation`).

## Preguntas frecuentes

**¿Puedo agregar una dependencia npm nueva?**
Solo con justificación en la PR y aprobación de un maintainer. El stack
está deliberadamente minimal (ver `AGENTS.md` "Lo que NO hacer"). No
agregamos Material, Bootstrap, Heroicons, Lucide ni librerías de state
management.

**¿Cómo pruebo cambios de SSR localmente?**
`npm run build && npm run serve:ssr:prime-showcase` levanta el server en
:4000. Después `npm run test:ssr:smoke` corre los 4 cookie cases del
dark-mode SSR. `npm run verify` hace todo end-to-end.

**¿Cómo refresco baselines visuales sin que CI falle por mi fuente local?**
Disparar manualmente el workflow `Visual baselines (manual)` en
`.github/workflows/visual-baselines.yml`. Corre en el mismo ambiente que
CI y commitea los snapshots actualizados a tu branch.

**¿Por qué `npm run lint` corre design-tokens:check?**
Para evitar drift entre `app.preset.ts`, `DESIGN.md` y
`design-tokens/tokens.json`. Si modificás el preset sin sincronizar,
`lint` falla local antes de que CI te pegue.

**¿El proyecto acepta cambios solo de docs?**
Sí, branch `docs/...` + PR con label `docs`. No requieren lint full
(solo markdown). Pero si modificás `DESIGN.md` o `AGENTS.md`,
verificá que no entren en conflicto con cambios en código abiertos en
otras PRs.
