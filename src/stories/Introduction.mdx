# PrimeNG Showcase — Design System

Este catálogo es el contrato visual de la aplicación. Cada primitive,
recipe y token surface vive acá y matchea exactamente lo que producción
renderiza.

## Idioma

El catálogo está en **español** porque es el idioma del equipo del
proyecto. El toolbar incluye un selector **🇪🇸 Español / 🇬🇧 English**
que cambia el contenido **interactivo** de las stories (labels de
botón, captions de demo) pero no la documentación estática — las
descripciones de los componentes son siempre español-primario. Ver
`.storybook/preview.ts → globalTypes.locale` para el detalle del
tradeoff.

Stories que opt-in a contenido bilingüe usan el helper
[`src/stories/i18n.ts`](https://github.com/floxcristian/prime-showcase/blob/main/src/stories/i18n.ts).

**Caso normal (90 % de las stories — solo español)**:

```ts
export const Primary: Story = {
  args: { label: 'Guardar' },
};
```

**Caso bilingüe** (cuando el demo sirve para validar copy localizado):

```ts
import { bilingualRender } from '../i18n';

export const Primary: Story = {
  render: bilingualRender(
    { label: { es: 'Guardar', en: 'Save' } },
    `<p-button [label]="label" />`,
  ),
};
```

## Cómo está organizado el catálogo

- **Tokens** — Colors, Typography, Spacing, Motion, Z-Index, Opacity.
  Los valores raw definidos en `src/app/app.preset.ts` (preset Aura)
  validados contra drift por `tools/design-tokens/sync.mjs`.
- **Primitives** — Stories de un solo componente con todos los
  argTypes configurados (Button, Card, Tag, Avatar, ListItem,
  EmptyState, DensityToggle). El reviewer cambia severity / state
  en el panel Controls y ve el resultado en vivo.
- **Recipes** — Composiciones canónicas extraídas de `DESIGN.md`
  (PageHeader, FormCard, MultiPanel, DataTable). Son los patrones
  que se copy-paste a los módulos; si un recipe acá derrapa, los
  módulos derrapan también.

## Cómo el design system se hace cumplir

Cuatro capas, de más fuerte a más débil:

1. **Tokens runtime** — `app.preset.ts` es la fuente única de verdad.
   El preset define el override Aura que PrimeNG resuelve en runtime.
2. **Reglas ESLint del design system** — 19 reglas custom en
   `tools/eslint/rules/*` bloquean violaciones en PRs (colores
   prohibidos, spacing off-scale, transition-all, etc.).
3. **Drift detection** — `npm run design-tokens:check` (parte de
   `lint`) asserts que `DESIGN.md` espeja `app.preset.ts`. Cualquier
   drift falla CI.
4. **Visual regression** — Playwright `toHaveScreenshot()` por ruta
   (`tests/visual/`) más captures a nivel Storybook planeados via
   Chromatic.
5. **Accessibility CI** — `@axe-core/playwright` escanea cada ruta y
   gate-ea en violaciones `serious`/`critical`. Mirrored en este
   catálogo via `@storybook/addon-a11y` (toggle el tab "Accessibility"
   en cualquier story).

## Trabajando con el catálogo

- **Toggle tema** — el icon sol/luna del toolbar aplica la clase
  `p-dark` al `<html>`, idéntico al app corriendo.
- **Toggle viewport** — el icon device del toolbar selecciona entre
  mobile (375), tablet (768), desktop (1280), wide (1440). Útil para
  validar recipes responsive.
- **Toggle idioma** — el flag 🇪🇸/🇬🇧 del toolbar cambia el contenido
  interactivo. La documentación estática queda en español.
- **Inspect accessibility** — el tab "Accessibility" en el addons
  panel muestra resultados de axe-core para la story actual.
  Failures acá fallarán también el workflow de a11y a nivel ruta.
- **Editar args en vivo** — el tab "Controls" deja cambiar props
  del componente renderizado. Útil para "¿cómo se ve este botón con
  severity=danger + outlined + loading?" sin escribir código.

## Dónde viven las cosas

| Archivo | Qué contiene |
|---|---|
| `DESIGN.md` | Token YAML front matter + recipes canónicas |

## Contribuir una story nueva

Una story nueva debe:

1. Vivir en `src/stories/primitives/<Component>.stories.ts` (o
   `recipes/`, `patterns/` según scope).
2. Usar `moduleMetadata` para importar el host/PrimeNG component.
3. Declarar cada `argType` significativo para que Controls funcione.
4. Cubrir al mínimo: estado default, cada severity/state usada en el
   app, y la variante disabled/loading cuando aplique.
5. Taggear con `'autodocs'` para que la Docs page se autogenere.
6. **Documentación en español** — descripciones, comentarios JSDoc,
   captions de templates. Si el contenido de demo tiene paridad
   bilingüe, usar `tr({ es, en }, ctx.globals?.['locale'])` (ver
   `Button.stories.ts → Primary` para el patrón canónico).
